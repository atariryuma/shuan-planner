# 週案プランナー エージェントガイド

このリポジトリを変更するエージェントは、着手前に次を読むこと。

1. [全体要件・プロダクト設計](docs/product-design-spec.md)
2. [UIテキスト規約](docs/ui-text-rules.md)
3. 印刷を変更する場合は [週案印刷の市場調査と設計方針](docs/print-market-research.md)
4. Google連携を変更する場合は [Google連携設定](docs/gas-setup.md)

## 最優先目標

先生が毎週、短時間で「作成、確認、印刷、提出、振り返り」を完了できる一般的な週案アプリにする。

機能数ではなく、次を優先する。

- 一週間を素早く俯瞰できる
- 編集対象へ少ない操作で到達できる
- 年間指導計画を再入力しない
- 時数の誤集計を防ぐ
- A4縦を標準に、提出可能な印刷物を安定して出す
- 担任、専科、複式で誤解のない集計軸を使う

## 実装判断の原則

- 一覧画面へ全文を常時表示しない。要約と詳細を段階表示する。
- 長文編集を狭い多列表へ押し込まない。
- モーダルは短い局所編集に限定する。大量・反復編集は専用画面で行う。
- カード内部の横スクロールを作らない。
- ページスクロールとモーダル内部スクロールを同時に強要しない。
- 破壊的操作以外は確認ダイアログよりUndoを優先する。
- 年間指導計画は表示時に解決し、週案へ内容を重複保存しない。
- 手動修正後も元の年間指導計画との参照を失わない。
- 専科の複数学級、複式の複数学年を単純合算しない。
- 印刷は罫線、ページ寸法、オーバーフローを実寸で検証する。
- 公開時は `sw.js` のバージョンを更新する。

## 変更時の最低確認

- 担任、専科、複式のいずれを壊していないか
- PC幅と狭い画面で操作できるか
- キーボードとタッチの両方で主要操作が可能か
- 保存、Undo、バックアップに影響しないか
- A4縦・横で文字切れ、罫線切れ、不要な空白ページがないか
- 既存データの移行と後方互換性が保たれるか
- 関連テスト、構文検査、`git diff --check` が通るか

大きな仕様判断は `docs/product-design-spec.md` を更新し、コードだけに知識を残さないこと。
